# MPI-169 — live RunPod API probe for volume used-bytes (2026-07-01)

Probed the real RunPod API (REST + GraphQL) through a temporary read-only route in the
running app (key via fork bridge; route reverted). DC = EU-RO-1.

## Verdict: RunPod exposes NO used-bytes for a network volume. DEFINITIVE.

- **REST `GET /networkvolumes`** → volume object keys = `['dataCenterId','id','name','size']`
  ONLY. The EU-RO-1 volume reads `size: 130` (GB total quota). No `used`, `free`,
  `consumed`, `percent`, `utilization` — nothing.
- **GraphQL `myself { networkVolumes { id name size dataCenterId } }`** → same four
  fields, resolves fine.
- **GraphQL field-name probe** `myself { networkVolumes { used usedBytes consumedBytes
  currentPerGBUsage } }` → each rejected:
  - `Cannot query field "used" on type "NetworkVolume"`
  - `Cannot query field "usedBytes" on type "NetworkVolume"`
  - `Cannot query field "consumedBytes" on type "NetworkVolume"`
  - `Cannot query field "currentPerGBUsage" on type "NetworkVolume"`

So the `NetworkVolume` GraphQL type has NO usage field, and REST returns only the quota.
This confirms the MPI-100 finding (downloadManager.js:1178-1185) at the API level:
**used-vs-quota is simply not exposed by RunPod.**

## Consequence

The progress-bar NUMERATOR does not exist server-side. The ONLY path to used bytes is to
measure it INSIDE a running Pod that has the volume mounted:
- Wrapper `GET /wrapper/disk` running `du -sb /workspace` (NOT statvfs — statvfs reads
  the multi-PB container overlay, was tried + reverted in MPI-100 image v0.4.5).
- => the bar can only reflect the volume of the DC whose Pod is CURRENTLY CONNECTED.
  An idle/selected-but-unconnected DC has no live reader → total-only badge (existing).

Denominator (`vol.size`, e.g. 130 GB) is already available from the volume object and
already shown in the settings badge (MpiSettings.js:1531).
