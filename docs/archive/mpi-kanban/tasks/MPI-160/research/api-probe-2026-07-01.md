# MPI-160 — live RunPod API probe (2026-07-01)

Probed the real RunPod GraphQL/REST API through a temporary read-only route in the
running app (key resolved via the fork bridge; route since reverted). DC = EU-RO-1.
No pod deployed — all deploy probes used deploy-proof inputs (bogus GPU / 9999GB RAM /
99 GPUs).

## Q1 — Does any query list per-host RAM configs? NO.

- `gpuTypes { lowestPrice(input:{dataCenterId:"EU-RO-1",secureCloud:true}) {...} }`
  for the RTX 5090 returns **one collapsed object**: `minMemory: 60, minVcpu: 16,
  uninterruptablePrice: 0.99, stockStatus: "Medium"`. Still a SINGLE scalar — no list.
- Probed extra `lowestPrice` fields `rentedCount`, `totalCount`, `rentalPercentage`
  — all **accepted by schema but return `null`**. RunPod does not populate host-count
  granularity. (Confirmed on MI300X too: same fields, all null.)
- `dataCenters { gpuAvailability {...} }` for EU-RO-1 5090 = **one row**:
  `{available:true, stockStatus:"Medium", displayName:"RTX 5090"}`. No RAM, no host
  split, no config count. `displayName`/`gpuTypeDisplayName` DO resolve (nice-to-have
  labels) but add no config granularity.
- `availableGpuTypes(...)` → **`Cannot query field "availableGpuTypes" on type "Query"`**
  — no such offerings/host-listing query exists.

**Verdict Q1:** there is NO RunPod query that enumerates the 60GB-vs-90GB host configs.
The "list every RAM config as its own row" design is **not achievable from a listing** —
the API collapses all hosts of a GPU/DC into one `minMemory` floor + one availability row.

## Q2 — Is `minMemoryInGb` a valid deploy input? YES (accepted) — but placement effect UNPROVEN.

- `podFindAndDeployOnDemand(input:{... minMemoryInGb: 90})` did **NOT** fail with an
  unknown-field / schema error. With a bogus `gpuTypeId` it failed only on
  `"Unknown GPU type"` (INVALID_INPUT) — i.e. the field passed schema validation and
  execution reached GPU resolution. **So `minMemoryInGb` is a real accepted input field.**
- Filter effect test was INCONCLUSIVE: valid 5090 + `minMemoryInGb: 9999` →
  `SUPPLY_CONSTRAINT "no longer any instances available"`. Control (valid 5090 +
  `gpuCount: 99`) → the **SAME** `SUPPLY_CONSTRAINT`. RunPod collapses ALL placement
  failures (impossible RAM, impossible GPU count, plain out-of-stock) into one generic
  supply error. So we **cannot tell from an error message** whether `minMemoryInGb`
  actually steers placement or is silently ignored.
- **Only a REAL deploy proves it:** `minMemoryInGb: 90` on an in-stock 5090 → let it
  create → read the landed pod's actual system RAM (wrapper /wrapper/stats). If it lands
  ≥90GB repeatably vs a control run without the field landing 60GB, the field works.
  Costs a real (brief, deletable) pod + a few minutes billing. Gated on user go-ahead.

## Consequence for the card

The original "list every config as a row" goal is **impossible as literally stated** (no
per-config listing). What IS potentially viable:

- **IF `minMemoryInGb` steers placement** (needs the live deploy proof): synthesize
  RAM-TIER rows in the picker (e.g. "RTX 5090 · ≥90 GB (LTX)" and "RTX 5090 · any RAM"),
  same price, and pass `minMemoryInGb` into the create spec (remoteProxy.js POST /pods
  and/or the GraphQL fallback input). Deterministic high-RAM landing = the real prize,
  BETTER than the old poll-and-hope lever B.
- **IF it's ignored:** fall back to the old lightweight levers — post-connect RAM warn
  via /wrapper/stats under ~90GB (lever A), optional wait-for-high-RAM poll on
  lowestPrice.minMemory (lever B; flat price = free to wait).

## Raw field reference (EU-RO-1, 5090, 2026-07-01)
- `lowestPrice` fields that EXIST: `minMemory, minVcpu, uninterruptablePrice,
  stockStatus, rentalPercentage(null), rentedCount(null), totalCount(null)`.
- `gpuAvailability` fields that EXIST: `available, gpuTypeId, stockStatus, displayName,
  gpuTypeDisplayName`.
- `NetworkVolume` fields that EXIST: `id, name, size, dataCenterId` ONLY (see MPI-169).
- `podFindAndDeployOnDemand` input `minMemoryInGb`: ACCEPTED (schema-valid).
- Placement errors are all generic `SUPPLY_CONSTRAINT` — not diagnostic.

# ADDENDUM — live deploy proof (2026-07-01, same session)

Ran the minMemoryInGb placement test for real. Temp route created ONE ephemeral 5090 in
EU-RO-1 per run (busybox, containerDisk 10GB), read the landed pod's system RAM
(machine.memoryInGb — the same field /remote/pod/specs trusts), then ALWAYS deleted it
(try/finally). Route reverted after; orphan sweep returned `reaped: []` (zero strays).

| minMemoryInGb | create result | landed RAM | machineId |
|---|---|---|---|
| (none, baseline) | ✅ created | 92 GB | d55y4zde8n4o |
| 16 | ✅ created | 92 GB | d55y4zde8n4o |
| 90 | ✅ created | 92 GB | d55y4zde8n4o |
| **200** | ❌ **FAILED — SUPPLY_CONSTRAINT** ("no instances available") | — | — |

All four runs hit the SAME DC within ~2 minutes. Baseline/16/90 all created fine; only
the impossible 200GB ask failed. The sole variable was the RAM value → **RunPod HONORS
`minMemoryInGb` as a hard placement FILTER.** Proven, not inferred.

## VERDICT: MPI-160 is FEASIBLE. The field works.

Nuances that shape the build:
- The filter is a FLOOR (`>= N GB`). "· ~90 GB (LTX)" row → send `minMemoryInGb: 90` →
  guaranteed a ≥90GB host or a clean SUPPLY_CONSTRAINT (retry/wait). This is the valuable
  row and it WORKS deterministically.
- CANNOT force a LOW config: a `minMemoryInGb: 60` floor still accepts a 90GB host, so a
  "· ~60 GB" row can't guarantee 60. But nobody wants to force low — the point is
  guaranteeing HIGH for LTX. So ship ONE meaningful extra row: a high-RAM "LTX" variant.
- At test time EU-RO-1 only had 90GB hosts free (baseline landed 92 with no ask), so the
  60-vs-90 SPLIT couldn't be exercised live — but that's irrelevant to the floor filter,
  which is proven.
- When the 90 floor can't be met (only 60GB hosts free), create fails with
  SUPPLY_CONSTRAINT → this pairs perfectly with the EXISTING auto-retry/wait-for-stock
  toggle (flat price = free to wait for a 90GB host to free up).

## RECOMMENDED BUILD (MPI-160)
Add a picker row/variant for the 5090 (and any LTX-relevant GPU): "RTX 5090 · ≥90 GB (LTX)".
Selecting it passes `minMemoryInGb: 90` into BOTH create paths:
- REST POST /pods spec (remoteProxy.js ~603) — add `minMemoryInGb` to spec.
- GraphQL fallback input (runpodRemote.js createPodGraphql ~175) — add
  `if (typeof spec.minMemoryInGb === 'number') input.minMemoryInGb = spec.minMemoryInGb;`
Wire the constraint through the connect flow + surface it in the picker. Pair with the
existing wait-for-stock toggle so a 90-floor miss waits instead of hard-failing. NOTE:
confirm REST POST /pods also accepts minMemoryInGb (only the GraphQL path was proven live;
the REST enum path may differ — quick REST probe before shipping).
