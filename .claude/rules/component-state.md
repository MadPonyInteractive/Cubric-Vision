## Sub-Agent Briefing
> Use this file when you need to know which state keys a component reads or writes.
> All keys live in `js/state.js` as a reactive Proxy. Writing any key auto-fires `state:changed`. Never manually emit `state:changed`.

> **Block-local (NOT in `state`):** `MpiGroupHistoryBlock` tracks the active tool mode in block-local variable `_options` (the currently-mounted `MpiToolOptions*` instance). This is intentionally NOT a `state` key — it is workspace-scoped and must not persist across navigation. Do NOT add an `activeTool` key to `state.js`.

> **Loop armed flag is session-only:** `state.loopArmed` controls whether the Cue dispatcher re-fires on queue drain. Never persisted to `project.json` or `modelSettings`. There is no "Single mode" — Cue is the only execution path; Loop is a flag layered on top. The Cue cluster (Cue/Stop/Clear) is always the same shape regardless of `loopArmed`; only the run button label + armed CSS class change.

> **`modelSettings[modelId]` shape (shared lifted 2026-05-22, schema v2).** Each model entry holds model-wide fields at the top level (`loras`, `upscaleModel`) and per-operation state under `operations: { [opName]: {...} }`. Cross-model shared state (ratio, batch, duration, motionIntensity, previewStage, qualityTier) now lives at `project.shared[mediaType]` where `mediaType` is `'image' | 'video'`. PromptBoxControls declare `scope: 'shared' | 'perOp'`: shared reads/writes via `getSharedSettings(project, mediaType)` / `setSharedSettings`; perOp via `getOpSettings(project, modelId, opName)` / `setOpSettings`. Helpers `_readSaved`, `_resolveDefault`, `_emitUpdate` in `PromptBoxControls.js` are the only API controls should use. See `.claude/rules/component-comfy.md` § "Persistence scope" for the full pattern. `MpiModelSettings` keeps writing `loras` and `upscaleModel` with the legacy `{ modelId, key, value }` payload (no `opName`) — projectService's back-compat guard routes those to the model bucket. Migration `v1 → v2` (`js/migrations/projectMigrations.js`) lifts legacy `modelSettings[*].operations.shared` to `project.shared[mediaType]` (last-wins per mediaType).

> **PromptBox reads `project.shared[mediaType].previewStage`** — boolean, persisted cross-model via `settings:shared:update` with `mediaType: 'video'` (the only `_ms` ops are video). Drives the "Preview initial stage" toggle (control id `previewStage`) and the `previewOnly` field on the run payload. Only present in operations whose `components[]` includes `'previewStage'` (today: `t2v_ms`, `i2v_ms`). `commandExecutor._buildParams` injects `Preview_Only` ONLY when the active op key ends in `_ms` — a stale toggle from a previous multi-stage op cannot leak into a single-stage workflow.

> **PromptBox `historyMode` is a context flag, NOT a state key.** `MpiGroupHistoryBlock` passes `historyMode: true` through `updateContext` whenever the video-history workspace mounts PromptBox. Effects: (1) PromptBoxControls hides + disables the `previewStage` control (prior persisted toggle value is preserved untouched for non-history contexts); (2) generationService / commandExecutor receive `historyMode: true` in the run payload and force `Preview_Only=false` for `_ms` ops regardless of the toggle; (3) `mpi-prompt-box--history-mode` root modifier class hides the media strip (chips still live in `_media`). No `state.js` key is added — workspace-scoped flags belong on the PromptBox context object.

> **Prompt drafts (`promptDraft` / `promptMedia`) are tagged single slots, NOT per-card maps (MPI-113).** Prompt text + staged input chips survive nav so a draft isn't lost switching gallery↔history. Each top-level slot (`gallery` / `history`) is keyed by `workspaceKey` and stamped with an `id` (the card id, via `workspaceId` prop). On mount PromptBox restores a slot ONLY when `(saved.id ?? null) === workspaceId` — else the previous card's draft would leak onto a different card. **Gallery** has no card (`workspaceId` omitted → `id:null`, always matches → persistent). **History** reuses the one `history` slot for every card, so only the LAST-TOUCHED card round-trips; opening any other card shows a clean box. Deliberately a single tagged slot — no per-card map, so no unbounded growth, no purge-on-delete hook. Session-only (cleared on restart). Two mount-order consequences: (a) restore runs INSIDE `MpiPromptBox.mount()`, BEFORE the parent block subscribes to `media-change`, so both blocks re-sync media counts / op-gating right after mount (`_syncBaseCtxFromPromptBox`+`_refreshOpOptions`+`_syncPromptToolDisabled` in history; `imageCount`/`videoCount`/`updateContext`/`refreshRadial` in gallery); (b) Reuse Prompt `clearMedia()`+re-injects AFTER mount, so a recall always wins over a restore. Settings stay per-model project state (`project.shared` / `operations[opName]`) — NOT part of the workspace draft. See `js/components/Organisms/MpiPromptBox/MpiPromptBox.js` (`_saveDraft`/`_saveMedia`/`_matchesSlot`).

> **Sidecar `extendedFrom: { id, displayName }`** is the only new history-item field added by the Continue-from-last-frame flow. Hydrated automatically by `projectReconciler` (pushes full sidecar meta). MpiHistoryList renders a 4th meta row `↳ <displayName>` when present. No new state key, no new top-level project field.

> **Sidecar `trim: { in: number, out: number } | null`** (video items only — added 2026-05-14, Phase D of video trim plan). Persisted by `MpiVideoControlBar` via `MpiGroupHistoryBlock` on `range-change` (debounced 250ms) through `POST /project-media/:projectId/update-meta` with `{ itemId, updates: { trim } }`. Route writes `Media/.meta/<uuid>.json` via the per-sidecar `updateItemMeta()` queue in `routes/projects.js` (atomic temp-rename, serialized per path). Absent / `null` ⇒ "no trim set" — control bar lazy-defaults to `[0, duration]` on each `loadedmetadata`. Restore path: caller passes `meta.trim` to `viewer.el.loadVideo(url, meta)`; viewer routes to control bar's `setPendingTrim(in, out)` (one-shot, applied on next `loadedmetadata`). Direct `setRangeQuiet` after `_setSrc` gets clobbered by the metadata reset. No new top-level state key. Range-aware ops (snapshot clamp, Continue from `out`, ffmpeg `-ss`/`-to` slice) consume this in Phase E.

> **Quality tier shares the `ratioSelector` key in the shared bucket** — control id `qualityTier` writes `project.shared[mediaType].ratioSelector.qualityTier`; the `ratio` control writes `selectedRatio` and `orientation` under the same `ratioSelector` object. One source of truth, two controls. Quality renders only when `RATIO_MODES[model.type] === 'quality'` (today `wan`; `ltx` future). Cross-control sync uses `Events.emit('ratio:quality-change', { modelId, qualityTier })` — the ratio control filters by `modelId` and calls `el.setQualityTier(tier)` to swap its ratio set in place. No direct DOM coupling between the two.

> **Cross-model slider controls persist under `project.shared.video`:** `duration` (int 1..30, default 5) for `t2v`/`i2v`/`t2v_ms`/`i2v_ms`; `motionIntensity` (float 0..1, default 0) for `i2v`/`i2v_ms`. Both written via `settings:shared:update` with `mediaType: 'video'` and recalled by `PromptBoxControls.mount()` via `getSharedSettings(project, 'video')`. Clamp + coerce on both save and recall — older builds may have written out-of-range values.

> **Per-op slider controls persist under `operations[opName]`:** `denoise` (float 0..1; default 0.20 for `upscale`, 0.30 for `detail` — defaults declared on the op via `commandRegistry.commands[op].defaults.denoise`), `useGrid` (boolean for `upscale`), `upscaleFactor` (float 1.5/2/3/4 for `upscale`). Independent value per op on the same model. Recalled by `PromptBoxControls.mount()` via `getOpSettings(project, modelId, opName)` (where `opName` is passed by `MpiPromptBox._refreshOpSlot`).

> **Resize tool settings persist under `project.toolSettings.resize`:** shape is `{ family: 'sdxl'|'flux'|'free', orientation: 'portrait'|'landscape', ratioLabel: string, multiplier: '1'|'2', width, height, upscale_method, keep_proportion, pad_color: { r, g, b }, crop_position, divisible_by, flip, rotation }`. `family` selects the resolution source: `sdxl`/`flux` derive `width`/`height` from `getModelRatios(family, orientation)` × `multiplier` (Width/Height inputs hidden); `free` uses manual Width/Height inputs. Whenever family/orientation/ratioLabel/multiplier change in preset mode, `width`/`height` are recomputed and persisted alongside. `MpiToolOptionsResize` reads them with `getToolSettings(project, 'resize', defaults)` and writes each control through `Events.emit('settings:tool:update', { toolKey: 'resize', key, value })`; `projectService` is still the sole writer to `project.json`. `getToolSettings(project, toolKey, defaults = { upscaleModel: null })` accepts tool-specific defaults so non-upscale tools do not inherit an upscale-only shape.

> **Upscale tool settings persist per-kind:** `project.toolSettings.imageUpscale` and `project.toolSettings.videoUpscale`, each shaped `{ factor: 'x1.5'|'x2'|'x3'|'x4', model: string }`. `MpiToolOptionsUpscale` is shared between image and video — `kind` prop selects the bucket. `model === ''` means "None" (run workflow with `Upscale_Using_Model: false`, no `Upscale_Model` injected). Both buckets persisted via `settings:tool:update` with `toolKey` = `imageUpscale` or `videoUpscale`.

> **Mask tool settings persist under `project.toolSettings.mask`:** shape is `{ model: string\|null, useBox: boolean, opacity: number, inverted: boolean }`. `model` is the auto-detector path (e.g. `bbox/face_yolov8n.pt`); `useBox` toggles Box vs Segment mode; `opacity` (0..1, default 0.7) controls the overlay alpha; `inverted` (default false) is a display-only flag — the underlying mask pixels are never mutated. `MpiToolOptionsMask` reads them with `getToolSettings(project, 'mask', defaults)` and writes each control through `Events.emit('settings:tool:update', { toolKey: 'mask', key, value })`. On mount the panel re-applies all four keys to `MpiCanvasViewer` (`setAutoMaskModel`, `setAutoMaskUseBox`, `setMaskOpacity`, `setMaskInverted`). The viewer caches `_isMaskInverted` on its own scope so the flag survives the canvas teardown/remount that `swapToPreview`/`swapToCanvas` performs.

> **Model LoRA settings shape:** most models, including LTX, use the flat shape `modelSettings[modelId].loras: Array<6>`. Models that declare `model.loraStages` (WAN) use a staged object, e.g. `loras: { high: Array<6>, low: Array<6> }`. `MpiModelSettings`, `commandExecutor`, and preview `loraSnapshot` handling must support both shapes.

> **Cue queue depth is local:** `state.generationQueueCount` includes the active Cue dispatch plus pending jobs. StatusBar subtracts the active dispatch and only displays pending depth, e.g. `GENERATING (2 queued)`. Do not poll ComfyUI queue depth for Cue mode.
> **Cue queue visibility comes from generationService:** user-facing queue surfaces must call `getGenerationQueueSnapshot()` or subscribe to `generation-queue:changed`. The snapshot owns stable `queueJobId`, running/pending status, model/operation, prompt excerpt, ratio/dimensions, batch count, loop source, stage label, media thumbnails, and cancel/stop affordances.
> **PromptBox ratio injection includes `Ratio_Label`:** the ratio control injects `Width`, `Height`, and selected `Ratio_Label`. Cue cards and sidecar metadata use the selected label, not a ratio inferred from pixel dimensions.

> **MpiCanvas pan/zoom is NOT in `state`:** `scale`, `offsetX`, `offsetY` live inside `ViewManager` (instance-local). Pan/zoom is applied as a CSS `transform` on `.mpi-canvas__stack` — not via `ctx.translate/scale`. Never reach into `state` for canvas view parameters.

> **Selected model is partitioned by mediaType:** `state.s_selectedModelIdByType = { image, video }`. Image and video selections persist independently — picking a video model never coerces the image slot and vice versa. Workspace Blocks read/write only the slot for their own mediaType. ALWAYS write via `setSelectedModelId(mediaType, id)` in `js/utils/modelHelpers.js` (top-level replace so the Proxy fires `state:changed`; also updates `s_lastSelectedMediaType`). Reading: `getSelectedModelId(mediaType)` or `resolveActiveModel(mediaType)` (with first-installed fallback). NEVER write `state.s_selectedModelIdByType[t] = id` directly — sub-object mutation is invisible to the Proxy. NEVER write at workspace mount time — the resolver already returns a valid fallback id; persisting it would clobber the sibling-type slot. Persisted to `localStorage` (`STORAGE_KEYS.SELECTED_MODELS` + `STORAGE_KEYS.LAST_SELECTED_MEDIATYPE`) by `state:changed` subscribers in `state.js`; hydrated at module init in `state.js`. Gallery is mediaType-agnostic — it reads `s_lastSelectedMediaType` on mount to decide which slot to restore.

> **MpiCanvas VRAM lifecycle:** When prompt tool is active, `MpiCanvasViewer` destroys `MpiCanvas` completely (`_cv.inst.el.destroy()` zeros canvas dims → immediate GPU texture release) and mounts `MpiMaskedImagePreview` (zero GPU backing). On switch back, a fresh `MpiCanvas` is remounted. The internal `_cv` mutable ref + `canvas` Proxy ensure all `canvas.*` calls in `MpiCanvasViewer` transparently forward to the current live instance. Do NOT hold a direct ref to the `MpiCanvas` instance from outside `MpiCanvasViewer` — the instance is replaced on every prompt↔tool swap.
